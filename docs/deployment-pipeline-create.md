# Set Up a GitHub Pipeline in AKS Desktop

This guide walks you through configuring a GitHub-based CI/CD pipeline for your AKS project using AKS Desktop. The pipeline uses GitHub Copilot Coding Agent to analyze your repository, containerize your application, and deploy it to Azure Kubernetes Service — all from within AKS Desktop.


> **Note:** This feature is currently in **Preview/Alpha** 

---

## What you'll get

By the end of this guide, your repository will have:

- A **Dockerfile** for your application (generated by Copilot)
- **Kubernetes manifests** (Deployment, Service, etc.)
- A **GitHub Actions workflow** that builds, pushes, and deploys your app to AKS on every merge
- **Workload identity federation** so GitHub Actions authenticates to Azure without storing secrets

---

## Prerequisites

### Azure

| Requirement | Details |
|---|---|
| **Azure subscription** | An active Azure subscription |
| **AKS cluster** | A running AKS cluster with **OIDC issuer** and **workload identity** enabled (see [Create a cluster](#create-an-aks-cluster-if-needed) below) |
| **Azure RBAC role** | You need **Owner** or **User Access Administrator** on the cluster's resource group to create role assignments for the managed identity |
| **Azure sign-in** | Signed in to Azure within AKS Desktop |

### GitHub

| Requirement | Details |
|---|---|
| **GitHub account** | With access to the repository you want to deploy |
| **GitHub Copilot** | A GitHub Copilot subscription that includes **Copilot Coding Agent** (GitHub Copilot Enterprise, or Copilot Individual/Business with agent access) |
| **Repository permissions** | **Admin** or **Write** access to the repository |

### AKS Desktop

| Requirement | Details |
|---|---|
| **AKS Desktop** | Latest version installed |
| **Cluster registered** | Your AKS cluster added to AKS Desktop |
| **Project created** | A project (namespace) created for your deployment |

## Create an AKS cluster (if needed)

If you don't already have a cluster with the required features, create one:

```bash
# Set your variables
RESOURCE_GROUP="my-resource-group"
CLUSTER_NAME="my-cluster"
LOCATION="eastus"

# Create resource group
az group create --name $RESOURCE_GROUP --location $LOCATION

# Create AKS cluster with required features
az aks create \
  --resource-group $RESOURCE_GROUP \
  --name $CLUSTER_NAME \
  --enable-oidc-issuer \
  --enable-workload-identity \
  --enable-azure-rbac \
  --enable-aad \
  --network-plugin azure \
  --generate-ssh-keys

# Get credentials
az aks get-credentials --resource-group $RESOURCE_GROUP --name $CLUSTER_NAME
```

If you have an existing cluster, you can enable the required features:

```bash
az aks update \
  --resource-group $RESOURCE_GROUP \
  --name $CLUSTER_NAME \
  --enable-oidc-issuer \
  --enable-workload-identity
```

> **Note:** OIDC issuer and workload identity are required for the pipeline to authenticate GitHub Actions with Azure securely, without storing long-lived credentials.

## Enable the Pipeline feature

The GitHub Pipeline feature is currently a **preview feature** that must be enabled in settings.

1. Open AKS Desktop
2. Go to **Settings** (gear icon in the sidebar)
3. Select **Plugins** > **aks-desktop**
4. Under **Preview Features**, toggle on **GitHub Pipelines**

## Step-by-step: Configure a pipeline

### Step 1 — Open the pipeline wizard

There are two ways to start:

- **From the project header:** Open your project, then click the **Configure Pipeline** button in the top-right corner
- **From the Deploy tab:** Open your project's **Deploy** tab, find the Pipeline card, and click **Configure Pipeline**

### Step 2 — Connect to GitHub

1. Click **Connect** to begin GitHub authentication
2. A browser window opens to GitHub's authorization page
3. Review the permissions requested:
   - **Contents** — push workflow and agent config files to your repo
   - **Workflows** — create and trigger GitHub Actions workflows
   - **Pull Requests** — create setup and deployment PRs
   - **Issues** — create tasks for the Copilot agent
   - **Actions** — monitor workflow runs
4. Click **Authorize** in the browser
5. Return to AKS Desktop — you should see **"Connected as \<your-username\>"**

### Step 3 — Select a repository

1. Browse or search for the repository you want to deploy
2. Click the repository to select it
3. AKS Desktop checks whether the **AKS Desktop GitHub App** is installed on the repository

**If the app is not installed:**

1. A prompt appears with an **Install GitHub App** button
2. Click it to open GitHub in your browser
3. Install the AKS Desktop app on the repository (or your organization)
4. Return to AKS Desktop — installation is detected automatically

Click **Next** to continue.

### Step 4 — Select a container registry (optional)

Choose how container images will be stored:

- **Select an existing ACR** — pick from the dropdown list of Azure Container Registries in your subscription
- **Create a new ACR** — enter a name (5–50 lowercase alphanumeric characters) and select a SKU tier (Basic, Standard, or Premium)
- **Skip** — proceed without a registry. You can configure one manually later, but the pipeline won't be able to build and push images until you do

### Step 5 — Set up workload identity

AKS Desktop automatically configures Azure workload identity so that GitHub Actions can authenticate to your AKS cluster without storing credentials.

1. Optionally edit the **Identity Resource Group** name (defaults to `rg-<project-name>`)
2. Click **Continue**
3. AKS Desktop performs these steps automatically:
   - Creates a resource group for the identity (if it doesn't exist)
   - Creates a user-assigned managed identity
   - Assigns the required Azure RBAC roles to the identity
   - Creates a federated credential linking the identity to your GitHub repository

You'll see a progress indicator for each step. When complete, the managed identity's **Client ID** is displayed.

> **What roles are assigned?** The identity receives roles scoped to your cluster and container registry (if selected). For standard namespaces: AKS Cluster User, AKS RBAC Writer, and ACR push permissions. For managed namespaces: AKS Namespace User, AKS RBAC Writer (scoped to the namespace), and ACR push permissions.

### Step 6 — Review agent configuration

Review the configuration summary before creating the setup PR:

- **Cluster**, **resource group**, and **namespace** for deployment
- **Application name** (editable — used for Kubernetes resource names)
- **Workload identity Client ID** (auto-populated from Step 5)
- **Container registry** (if selected in Step 4)

Expand **Container Deployment Settings** to optionally configure:
- Container port
- Service type (ClusterIP or LoadBalancer)
- Environment variables

When ready, click **Create Setup PR**.

### Step 7 — Review and merge the setup PR

AKS Desktop creates a pull request in your repository with two files:

| File | Purpose |
|---|---|
| `.github/workflows/copilot-setup-steps.yml` | Agent environment setup workflow |
| `.github/agents/containerization.agent.md` | Instructions for the Copilot agent, including your AKS configuration |

1. Click **Review on GitHub** to open the PR
2. Review the files and merge the PR
3. AKS Desktop automatically detects the merge and proceeds to the next step

### Step 8 — Copilot agent analyzes your repository

After the setup PR is merged, AKS Desktop triggers the GitHub Copilot Coding Agent. The agent:

1. Analyzes your repository's code and structure
2. Generates a `Dockerfile` tailored to your application
3. Creates Kubernetes manifests (`Deployment`, `Service`, etc.)
4. Creates a GitHub Actions deployment workflow (`.github/workflows/deploy-to-aks.yml`)
5. Opens a new **deployment PR** with all generated files

This typically takes a few minutes. AKS Desktop shows progress in real time.

### Step 9 — Review and merge the deployment PR

1. Click **Review on GitHub** to open the deployment PR
2. Review the generated files:
   - `Dockerfile` — verify the build is correct for your app
   - Kubernetes manifests — check resource limits, ports, and configuration
   - Deployment workflow — review the CI/CD pipeline steps
3. Request changes if needed, or merge the PR

### Step 10 — Deployment

Once the deployment PR is merged:

- The `deploy-to-aks.yml` workflow runs automatically
- It builds your container image, pushes it to the registry, and deploys to AKS
- AKS Desktop shows the deployment status in the wizard

When complete, your application is running on AKS.

## Redeploying

After the pipeline is configured, you can trigger new deployments from the **Deploy** tab:

1. Open your project's **Deploy** tab
2. Find your deployment in the workloads list
3. Click **Deploy via Pipeline**
4. Confirm the deployment parameters
5. Click **Deploy** to trigger the workflow

New deployments are also triggered automatically when you push changes to the repository and merge to the default branch.

## What gets created in Azure

The pipeline setup creates these Azure resources:

| Resource | Name pattern | Purpose |
|---|---|---|
| Resource group | `rg-<project-name>` | Contains the managed identity |
| Managed identity | `id-<project-name>-github` | Authenticates GitHub Actions to Azure |
| Federated credential | On the managed identity | Links GitHub OIDC tokens to the Azure identity |
| Role assignments | On cluster and/or ACR | Grants the identity permission to deploy |

---

## What gets created in GitHub

| Resource | Details |
|---|---|
| Repository secrets | `AZURE_CLIENT_ID`, `AZURE_TENANT_ID`, `AZURE_SUBSCRIPTION_ID`, and `AZURE_ACR_NAME` (if ACR selected) |
| Workflow files | `.github/workflows/copilot-setup-steps.yml` and `.github/workflows/deploy-to-aks.yml` |
| Agent config | `.github/agents/containerization.agent.md` |

## Troubleshooting

### "The AKS Desktop GitHub App must be installed"

The GitHub App is required for AKS Desktop to interact with your repository. Click the provided installation link and install the app on the repository or organization.

### Workload identity setup fails

- **Permissions error:** Ensure you have **Owner** or **User Access Administrator** role on the cluster's resource group
- **OIDC issuer not enabled:** Run `az aks update --resource-group <rg> --name <cluster> --enable-oidc-issuer --enable-workload-identity`
- **Resource provider not registered:** Run `az provider register --namespace Microsoft.ManagedIdentity --wait`

### Copilot agent doesn't respond

- Verify your GitHub account has access to **Copilot Coding Agent**
- Check that the Copilot agent is enabled for your repository in GitHub settings (Settings > Copilot > Coding agent)
- Review the agent task issue on GitHub for error details

### Deployment workflow fails

- Check the GitHub Actions run logs for specific errors
- Verify the AKS cluster is running: `az aks show --resource-group <rg> --name <cluster> --query powerState`
- Verify the managed identity still has the correct roles: `az role assignment list --assignee <client-id> --all -o table`
- If the ACR push fails, verify the identity has `AcrPush` role on the registry

### Pipeline wizard is stuck or polling

- The wizard automatically polls GitHub for PR and workflow status (up to 1 hour for PRs, 30 minutes for workflows)
- If stuck, click **Review on GitHub** to check the status directly
- Use **Start Over** to reset the wizard and begin fresh

## Cleaning up

To remove pipeline resources:

```bash
# Delete the identity resource group (contains the managed identity)
az group delete --name rg-<project-name> --yes

# Remove GitHub repo secrets (optional)
gh secret delete AZURE_CLIENT_ID --repo <owner>/<repo>
gh secret delete AZURE_TENANT_ID --repo <owner>/<repo>
gh secret delete AZURE_SUBSCRIPTION_ID --repo <owner>/<repo>
gh secret delete AZURE_ACR_NAME --repo <owner>/<repo>
```

You can also delete the workflow files and agent config from your repository if you no longer need the pipeline.
